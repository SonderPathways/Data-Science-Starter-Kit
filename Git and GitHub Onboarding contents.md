### 4. The Problem Without Version Control

- `final_v3_really_final.xlsx`  
- Multiple email attachments  
- Hard-to-track changes  
- File conflicts  
- No ability to revert to earlier versions

### 5. What Is Version Control?

A system that:

- Tracks changes over time
- Restores old versions  
- Enables safe experimentation  
- Allows parallel collaboration  
- Improves transparency & reproducibility  

### 6. Why Version Control Matters for Data Science

- Traceability of analyses and models  
- Reproducibility  
- Safe experimentation using branches  
- Collaboration across distributed teams  
- Documentation of decisions

### 7. Why Version Control Matters for Designers

- Versioning of Illustrator & Figma files  
- Safe sharing of editable assets  
- Ability to rollback changes  
- Track contributions  
- Collaborate asynchronously

### 8. Key Concepts

- **Repository**: Project folder tracked by Git  
- **Commit**: Snapshot of changes  
- **Branch**: Parallel working line  
- **Merge**: Integrate branches  
- **Remote**: Cloud copy (GitHub)

### 9. Git vs GitHub

| Git | GitHub |
|-----|---------|
| Local tool | Cloud platform |
| Tracks file changes | Collaboration hub |
| Works offline | Issues, PRs, Projects, Permissions |

### 10. High-Level Git Workflow

1. Make changes  
2. Commit changes  
3. Push to GitHub  
4. Open Pull Request  
5. Review & merge  
6. Update main project

### 11. Example Data Science Repository Structure

Starter Kit Training Series – Session 2

### 6. Making Your First Commit

1. Edit file  
2. Write commit message  
3. Commit  
4. Push  

### 7. Writing Good Commit Messages

Good examples:

- `Add preprocessing for missing values`  
- `Fix color scale in map visualization`  
Rules:
- Be concise  
- Use the imperative  
- Explain “why”, not just “what”

### 8. Branching Strategy

- `main`: stable  
- Feature branches for new work  
Examples:
- `feature/add-first-notebook`  
- `design/update-icons`  
- `fix/typo-readme`

### 9. Creating a Branch in GitHub Desktop

1. Branch → New Branch  
2. Name it  
3. Work safely  

### 10. Opening a Pull Request (PR)

- Submit changes for review  
- Discuss improvements  
- Merge when approved  

### 11. Anatomy of a Pull Request

- Title  
- Description  
- Linked Issue  
- Files changed  
- Reviewer comments  
- Merge button  

### 12. What Reviewers Look For

- Clear & clean changes  
- Documentation  
- Reproducible analysis (DS)  
- Source files included (design)

### 13. Using Issues

Track:

- Bugs  
- Enhancements  
- Tasks  
- Questions  
Use labels for organization.

### 14. Project Boards (Optional)

Kanban view:

- To Do  
- In Progress  
- Review  
- Done  

### 15. Best Practices for Teams

- Use branches  
- Keep PRs small  
- Commit often  
- Write documentation  
- Avoid raw data >100MB  
- Designers: always include sources

### 16. Best Practices for External Contributors

- Fork first  
- Follow naming conventions  
- Use Issues for questions  
- Respect Code of Conduct
